# Adding trips to the review queue — the well-trodden path

How a trip appears in the review app for a reviewer, what has to be true for it to be
reviewable, and exactly which repo owns each step. This is the single reference for
**both new trips (straight off the pipeline) and legacy/off-pipeline batches**.

> **Two repos, one queue.** The review app (this repo, `dynamic-review`) only ever
> *reads* the queue. The queue itself is produced entirely by the content-pipeline repo
> (`dynamic-content`, the `Scripts` tree). So "add a trip to review" is almost always
> **Scripts-side work**; the review app just has to be able to *resolve* the trip's
> audio, images, voice and language. The Scripts-side half of this contract lives at
> `Scripts/Trello/REVIEW_QUEUE_HANDOFF.md` — keep the two in sync.

---

## 1. What the queue actually is

- `GET /api/trips` reads **one file**: `trips_to_review.json` (repo root,
  `config.MANIFEST_PATH`). There is **no** second/override manifest, and hand-editing it
  is pointless — it is regenerated wholesale on every export.
- That file is generated by **`Scripts/Trello/export_review_trips.py`**, which reads
  **live Trello lists 6 (Human Translator Review) and 7 (Human KP Confirm)** and expands
  each card's `[review]` block into one manifest row per rung. The export **commits +
  pushes** the manifest from wherever it is run (normally the workstation).
- The live app on the **Ubuntu laptop** serves whatever manifest it has pulled — so the
  last step of any queue change is **`git pull` on the laptop**.
- Reviewer order = `(lane, Trello card position, trip_id)`. Drag a card in Trello to
  reorder; the in-app admin **pin** floats a trip above that at display time.

So the durable mechanism is always: **Trello card + `[review]` block → export →
laptop pull.** Everything below is about making those three steps produce the right rows.

---

## 2. The reviewability contract (what the review app needs)

A row in the manifest only becomes a *usable* review if the app can resolve the trip.
For each rung the app needs **all** of:

| # | Requirement | Where it lives | Notes |
|---|---|---|---|
| A | A **staging Firebase `Trips/{cid}`** doc | staging Firestore | The source of truth for text/scenes. `reviewable` and the editor both read it. |
| B | **Scene audio** `<i>.mp3` (+ `<i>_q.mp3` keyword, `<i>_a*.mp3` options) | R2 `review-audio/<cid>/` **or** a local master tree | R2 is sufficient — no local master needed. The backend lazily mirrors R2 → `work/_r2_seed_cache/<cid>/`. Gate: at least one bare `<i>.mp3`. |
| C | A resolvable **voice** | manifest `voice` → `voice_overrides.json` → staging voice → registry → `LANGUAGE_FALLBACK_VOICE` | For ES the fallback is already `sara`, so the manifest `voice` field is optional. |
| D | **Scene images** (optional but wanted) | R2 `dynamic-languages-thumbs` | See §6. A trip is reviewable without images; scenes just render "no thumbnail". |
| E | A reviewer with the **language ACL** | `users.languages_json` on the laptop DB | The ACL value is the **language name** (`"Spanish"`, `"French"`…), **not** a code like `"es"`. Set with `manage.py set-languages`. |

Voice / speed / language resolution is **fully automatic from the id suffix** — no code
change per trip:

- `speed_for_trip`: `_A12_ES`/`_Beg_ES` → **0.7×**, `_B1_ES` (EU) → **0.85×**,
  `_B2_ES`/native → 1.0×. Same for FR/DE/IT. (CJK is handled separately.)
- `language_of`: `_ES` → `"Spanish"` (and `_FR/_DE/_IT`, `_JP`, `_ZH`, `_EN`).
- Bare `<Family>_A12_ES` ids resolve correctly **without** a `_Beg` segment.

If you are adding a language/level the app has never seen, check that
`audio_core` (`_EU_LANG_SUFFIXES`, `speed_for_trip`, `VOICE_REGISTRY`,
`LANGUAGE_FALLBACK_VOICE`) and `manage.py VALID_LANGUAGES` already know it — otherwise
that is a genuine code change, not a data task.

## 3. Producer path A — new trips off the standard pipeline

This is the common case: a family already has a Trello card (created in the early
pipeline lanes) and its target-language audio has just been produced at lane **5c**.
The card, the data dir + `source.json`, and the TripGroup all already exist.

On the **workstation** (`D:\Dynamic Languages\Scripts`):

```bash
# 5c already uploaded the audio to R2 review-audio/<cid>/ (run_levels.py / run_5a.py /
# run_n4.py / run_n5.py do this in-process). If you produced audio some other way:
py -3.12 upload_review_audio_r2.py --manifest        # fill any R2 gaps

# 1. Write/refresh the family card's [review] block + wire the variant's TripGroup entry
py -3.12 Trello/sync_review_variant.py --variant <cid> --voice <Voice> --apply

# 2. Regenerate + commit + push the manifest
py -3.12 Trello/export_review_trips.py
```

Then on the **laptop**: `git pull` the review-app repo.

`sync_review_variant` is the canonical 5c write path: it rebuilds the block from the
family's audio-present variants and ensures TripGroup membership. `audio_present`
counts **R2** audio, so a variant with no local master still lists.

## 4. Producer path B — off-pipeline / legacy batches

When drafts did **not** come through the standard pipeline they can be missing the two
things discovery depends on: a **`source.json`** in the data dir, and a **Trello card**.
Symptoms: the export's `audit_unlisted` never mentions them, `make_eu_review_cards`
skips them, and a hand-made card lands them in **lane 7 with the wrong family**.

Root cause: under `review_block.STRICT_DATA_ROOTS`
(`Scripts/Research and Writing/data`), a dir is admitted to the variant index **only if
it carries `source.json`** — the name-derived fallback is deliberately off there, and is
also off for EU `_A12_/_B1_` ids. No `source.json` ⇒ `source_en_id_of(cid)` returns the
cid itself ⇒ `is_native` true ⇒ forced to lane 7.

**Fix (Scripts-side):**

1. **Write `source.json` to exactly the rungs you want reviewed** — and nothing else.
   Value = the rung's real native parent id (`source_en_id`). This is the single control
   that (a) puts them on lane 6, (b) sets a clean `family`, and (c) guarantees *only*
   those rungs list (discovery is dir-based, not audio-based). For the gallery batch the
   correct value was each draft's own `draft.json["source"]` (`<X>_Trip`); mirror whatever
   the real parent is for your batch, or the existing `_Beg`/`_Beg_ES`/`_Int` dir if one
   exists. Do **not** invent a parent id that points at nothing.
2. **Create the lane-6 cards:** `py -3.12 Trello/make_eu_review_cards.py --list 6 --only <cids…> --apply`
   (idempotent; `--only` keeps it off already-carded families). This writes a bare
   `[cid:<sid>]` card **with no `[review]` block yet.**
3. **Populate the `[review]` blocks — Trello only:**
   `py -3.12 Trello/backfill_review_blocks.py` (dry-run), then `--apply`. **Do not** use
   `make_eu_review_cards --sync` / `sync_review_variant` unless you intend to write
   TripGroup membership into staging Firebase (they call `ensure_membership`).
   ⚠️ **Gate the dry-run:** it must change **only your new cards.** Row *removals* on any
   other lane-6/7 card mean R2 was unreachable mid-run — `backfill` has no
   `preserve_existing_rows` guard (`sync_review_variant` does), so a transient
   `audio_present=False` drops that rung; abort and re-run once R2 is up.
   ⚠️ **Run this before any `export`** — a card with a bare `[cid:]` and no block yet
   exports as a lane-7 native row via the legacy fallback.
4. **Export:** `py -3.12 Trello/export_review_trips.py`.
5. **Laptop:** `git pull` the review-app repo.

A fully worked instance of this path (the 35 Spanish gallery rungs, 2026-07-22) is the
reply prompt at `Scripts/Research and Writing/Gallery Spain/REVIEW_APP_REPLY.md`.

## 5. Review-app operator steps

These are the only steps that happen **in this repo / on the laptop**:

- **Provision the reviewer's language ACL** (once per language per reviewer), on the
  laptop: `py -3.12 backend/manage.py set-languages --username <who> --languages Spanish`
  (comma-separated for several). Admins see everything; reviewers only see trips whose
  `language_of` is in their `languages_json`. **Use the language name, not a code.**
- **Deploy any app code change** a batch needed (rare): `git pull` +
  `systemctl restart review-app.service` in an idle window; FE changes need
  `npm run build`. Never restart / write `review.db` while a reviewer is editing.
- **Verify:** open one trip from the queue in the app — it must load its staged text,
  play audio, and (if images were supplied) show scene thumbnails.

### 5b. ⚠ When the producer CHANGES a trip that is already in the queue

Not a new trip — a re-publish + re-upload of a queued one (e.g. a remediation batch).
Adding audio to R2 and text to staging is **not** enough.

**Use the committed tool** (on the laptop, from the review-app root, Scripts venv):

```bash
py -3.12 scripts/backup_review_db.py backup            # always first
py -3.12 scripts/refresh_trips.py audit  --file cids.txt   # verdict per trip
py -3.12 scripts/refresh_trips.py clear  --file cids.txt
#   … load the trip list in the app so the caches refill, then:
py -3.12 scripts/refresh_trips.py verify --file cids.txt   # cached bytes == R2?
py -3.12 scripts/refresh_trips.py reseed --file cids.txt   # only if audit said RESEED
```

`audit` classifies each trip **CLEAR** (no session — the cache clear is all it needs),
**RESEED** (a session with no work product) or **HANDS OFF** (live reviewer, reviewer work,
a non-`in_review` status, or a completed trip that belongs on the delta path). `reseed`
re-checks every guard and aborts rather than touching a trip it shouldn't.

**Order matters:** run this AFTER the producer's uploads land. Clearing first just
re-caches the old audio.

What it is doing, and why it can't be skipped:

- **The R2 seed cache.** `sessions.resolve_audio_dir` mirrors `review-audio/<cid>/` into
  `work/_r2_seed_cache/<cid>` and returns that folder on every later call — it **never
  re-downloads**. It is filled by the trip **listing** (`sessions.py`, the `reviewable`
  probe), not just by opening a trip, so on the laptop practically every queued trip is
  already cached. Fix per changed cid:
  `rm -rf backend/work/_r2_seed_cache/<cid>` — no restart needed (`review_audio` keeps no
  listing cache). Skipping this serves the reviewer NEW text over OLD audio.
- **Any session that predates the change.** Staging text is read live at **seed**, so a
  trip with no session picks the new text up by itself; an open session froze
  `original_text` and must be deleted to re-seed. Check `presence` first — never re-seed a
  trip a reviewer is actively editing — and check `field_edits` for corrected takes
  (`working` ≠ `orig`) before deleting: a correction is only safe to discard if R2 already
  holds it (combine-time mirror) or the text it voices no longer exists.

**The clear is not durable — the cache refills.** The trip listing re-downloads it from R2
within minutes of the clear, which is fine (R2 now holds the new audio) but means "the
cache is gone" is NOT the success condition. **"The refilled cache matches R2
byte-for-byte" is** — that's what `verify` checks (MD5 via ETag, no downloads).

**A cache↔R2 difference is not always staleness.** If R2 also holds
`review-audio/<cid>/originals/<name>`, that clip is a reviewer's CONFIRMED take — the app
archives the as-delivered master there when a correction overwrites it, so R2 is *ahead* of
the cache by design. `verify` checks that marker and reports such clips as expected rather
than as a failed refresh.

Approve-time safety, for reassurance: a stale session cannot blanket-clobber the producer's
upload — only fields whose working take differs from the pristine seed are promoted and
mirrored. The narrow risk is a *corrected* take whose text was rewritten upstream.

Worked examples: 2026-07-23 — 50 EN trips, then 12 FR trips (see that day's session log).

## 6. Scene images

Two independent image paths feed the SceneCard (`thumb_url ?? image_url`):

- **VID scenes** → `thumb_url` via `thumbs.py`: Vimeo id → stem (from `VideoIds-*.json`)
  → R2 `dynamic-languages-thumbs/scene-thumbs/<stem>.jpg`. This only works if the id is
  in the VideoIds snapshot **and** the JPG is on R2 — often false for legacy/gallery
  video.
- **Static / fallback** → `image_url` via the overlay resolver: R2
  `dynamic-languages-thumbs/review-overlays/<cid>/<i>.jpg`, served at
  `https://thumbs.dynamiclanguages.org/review-overlays/<cid>/<i>.jpg`, keyed by the exact
  rung id and scene index (`_overlay_base` applies no reduction to `_ES` ids).

Since 2026-07-22 a **VID scene whose `thumb_url` doesn't resolve also falls back to
`image_url`** (`sessions.get_session`). So the robust, uniform convention for any batch
whose Vimeo ids aren't in the VideoIds snapshot is: **the producer uploads each scene's
still to `review-overlays/<cid>/<i>.jpg`**, one folder per rung id, where **`<i>` is the
scene's index in the staged `Trips/<cid>.quickTrips` array** — not any capture/source
numbering. A wrong index shows the wrong or a blank still with no error, so spot-check one
scene in-app. Images resolve at serve time — **no re-export needed**, they appear once
uploaded. A trip is reviewable without them.

## 7. Cheat sheet

```
Queue = trips_to_review.json  ⇐ export_review_trips.py ⇐ Trello lanes 6/7 [review] blocks
Reviewable = staging Trips/{cid}  +  R2 review-audio/<cid>/<i>.mp3  +  reviewer ACL=<Language>
Images     = R2 dynamic-languages-thumbs/review-overlays/<cid>/<i>.jpg  (per scene, optional)
New trip   : 5c → sync_review_variant --variant <cid> --apply → export → laptop git pull
Off-pipeline: source.json(exactly the rungs) → make_eu_review_cards --apply
             → backfill_review_blocks --apply (NOT --sync) → export → laptop git pull
ACL        : backend/manage.py set-languages --username <who> --languages Spanish
```

Scripts-side contract: `Scripts/Trello/REVIEW_QUEUE_HANDOFF.md`.
